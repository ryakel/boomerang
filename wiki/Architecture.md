# Architecture

## Overview

```
Browser (React PWA)
  ├── localStorage (fast cache)
  └── fetch /api/* ──> Express Server
                          ├── /api/data/*      → SQLite (persistence)
                          ├── /api/messages    → Anthropic Claude API
                          ├── /api/notion/*    → Notion API
                          └── /api/keys/status → Reports env var availability
```

## Data Flow

1. **On app load**: Frontend fetches `GET /api/data` → hydrates localStorage from SQLite
2. **During use**: All reads come from localStorage (instant). All writes go to localStorage AND debounced `PUT /api/data` to SQLite.
3. **API calls**: Frontend sends requests to `/api/*`. Server resolves API keys from request headers (user-provided) or env vars (server default), then proxies to external APIs.

## Storage

- **localStorage** (`boom_tasks_v1`, `boom_routines_v1`, `boom_settings_v1`, `boom_labels_v1`) — browser-side cache for fast reads
- **SQLite** (`/data/boomerang.db`) — single table `app_data` with collection name as PK and JSON blob as value. Source of truth for persistence.

## API Key Resolution

```
Request arrives
  → Check x-anthropic-key / x-notion-token header (user-provided via UI)
  → Fall back to ANTHROPIC_API_KEY / NOTION_INTEGRATION_TOKEN env var
  → If neither: return 400 with helpful error
```

Keys set in the UI are stored in localStorage settings and sent as request headers. They never touch the server's filesystem or database — the server only forwards them to the external API.

## PWA

The app is a Progressive Web App:
- Service worker (generated by vite-plugin-pwa) precaches all static assets
- Installable to mobile home screen
- Works offline for viewing/editing tasks (API features require connectivity)
- Auto-updates when new versions are deployed
